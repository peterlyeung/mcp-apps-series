Claude Desktop & MCP

Three related but distinct capabilities

Episode 03 — mcp-apps-series

Claude Desktop has at least three MCP-adjacent capabilities that are easy to lump together but are actually separate layers. Getting this distinction right matters, because it's the part most likely to confuse people trying this out for the first time.

Three layers

1. MCP servers — the base protocol

Local (stdio) or remote — gives Claude tools, resources, and prompts from that server.

2. MCPB — desktop extensions

How you install a local MCP server: a zip file, one-click setup.

3. MCP Apps — interactive UI rendering

What happens after a tool call, if the tool declares a UI resource.

Layer one is the base protocol — connecting to an MCP server at all, local or remote, to get its tools, resources, and prompts. Layer two, MCPB, is specifically about installation — packaging a local server as a one-click extension. Layer three, MCP Apps, is about what happens after a tool call — whether the result renders as an interactive UI. These three are independent axes, not steps in a single pipeline.

MCPB — desktop extensions

  • An .mcpb file = a zip archive with a local MCP server + a manifest.json
  • One-click install: double-click, drag in, or Settings → Extensions
  • Runs locally over stdio; bundles dependencies; works offline; no OAuth needed
  • Good fit for firewalled systems, local tools, piggybacking on existing SSO
  • Previously called "DXT" — renamed to MCPB
MCPB — an m c p b file is a zip archive containing a local MCP server plus a manifest dot json. You install it with a double-click, by dragging it in, or through Settings, Extensions. It runs locally over stdio, bundles its own dependencies, works offline, and needs no OAuth. Good fit for internal tools behind a firewall, or anything that wants to piggyback on existing single sign-on. This used to be called D X T, and got renamed to MCPB.

MCP Apps support in Claude Desktop

  • Claude — both claude.ai web and Claude Desktop — is a launch host for MCP Apps, since Jan 2026
  • Any connected tool that declares a ui:// resource renders as a sandboxed iframe, inline in chat
  • Works the same whether the server was installed via MCPB or connected remotely — install and render are orthogonal
Known rough edge: there's at least one open GitHub issue about MCP Apps UI not always rendering in Claude Desktop / claude.ai. Bleeding edge — expect occasional bugs.
Claude — both the web app and Claude Desktop — is a launch host for MCP Apps. Any connected tool that declares a ui resource renders as a sandboxed iframe right inline in the chat, and this works identically whether the server was installed as an MCPB extension or connected remotely — install and render are genuinely orthogonal concerns. One honest caveat: this is bleeding edge, there's at least one open GitHub issue about rendering not always working — worth expecting occasional bugs at this stage.

How to actually try this today

  1. Settings → Extensions in Claude Desktop
  2. Browse extensions (Anthropic-reviewed) or install a custom .mcpb
  3. Call a tool that declares an MCP Apps UI resource in a normal conversation
  4. For remote servers, same rendering applies once connected — no MCPB needed
If you want to try this yourself: open Settings, then Extensions, in Claude Desktop. Browse the Anthropic-reviewed directory, or install a custom m c p b file. Then just call a tool that declares an MCP Apps UI resource in a normal conversation — that's it, no extra steps. And for remote MCP servers, the exact same rendering applies once you've connected them — you don't need MCPB at all for that path. Next episode, we stop describing this and actually build one.
