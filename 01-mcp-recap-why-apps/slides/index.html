MCP Recap & Why Apps

The gap MCP Apps was built to close

Episode 01 — mcp-apps-series

Before we get into MCP Apps specifically, a quick recap of MCP itself, and the actual problem that led to Apps getting added as an extension in the first place. If you already know MCP well, this is a fast one.

MCP, in one paragraph

MCP is a JSON-RPC based protocol connecting an LLM host — Claude Desktop, an IDE, whatever — to external servers.

Servers expose three primitives: tools the model can call, resources the host can read, and prompts — reusable templates.

Before MCP Apps: everything a tool returned was text, structured JSON, images, or links — rendered as plain conversational content.

MCP is a standard protocol for connecting an LLM host to external servers. Those servers expose three things: tools the model can call, resources the host can read, and prompts — reusable templates. Up until MCP Apps, whatever a tool returned was always rendered as plain conversational content — text, JSON, images, links. Nothing interactive.

The gap: text breaks down for...

  • Exploring data — a wall of numbers vs. a map you click and drill into.
  • Configuring many options — ten back-and-forth questions vs. one form with defaults.
  • Rich media — a PDF or 3D model needs an actual viewer, not a description.
  • Real-time monitoring — a dashboard needs to update live, not on request.
  • Multi-step workflows — approvals and triage need navigation and persistent state.
Five situations where text-only responses genuinely break down. Exploring complex data — a wall of numbers versus an interactive map. Configuring lots of options — ten back and forth questions versus one form. Rich media like PDFs or 3D models that need an actual viewer. Real-time monitoring, where a dashboard needs to update on its own. And multi-step workflows like approvals, which need real navigation and state that persists across turns.

Why not just send a link?

Context preservation

No tab switching — the UI lives right inside the conversation.

Bidirectional data flow

The app can call any tool on the server; no separate API or auth stack.

Host delegation

The app can ask the host to act — the host routes it through tools the user already connected.

Security by construction

Runs in a sandboxed iframe — the host can safely render code it doesn't fully trust.
A reasonable question: why not just build a normal web app and send a link? Four reasons MCP Apps beats that. Context preservation — no tab switching, the UI is right there in the thread. Bidirectional data flow — the embedded app can call tools on the same server for free, reusing MCP's existing session, instead of standing up a separate API. Host delegation — the app can ask the host to take an action, and the host routes it through whatever the user has already connected. And security by construction — it runs in a sandboxed iframe, so the host can render it without fully trusting the server author.

MCP Apps was co-developed by Anthropic and OpenAI, building on the community MCP-UI project, and shipped as SEP-1865 — the first official MCP extension — on January 26th, 2026. Launch host support included Claude on web and desktop, ChatGPT, VS Code Copilot, and Goose, with launch partners like Figma, Canva, Slack, and Salesforce.
