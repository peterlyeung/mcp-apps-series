MCP Apps Deep Dive

Protocol, lifecycle, and the security model

Episode 02 — mcp-apps-series

Now the part most people skip past: how MCP Apps actually works under the hood. The protocol messages, the exact lifecycle from tool call to rendered UI, and the security model that makes it safe for a host to render code from a server it doesn't fully trust.

The core pattern: two primitives combined

A tool
Its description includes _meta.ui.resourceUri, pointing at a ui:// resource
A UI resource
An HTML page, usually bundled with its own JS and CSS
Everything comes down to two existing MCP primitives combined in a new way. A tool whose description includes a meta dot ui dot resourceUri field, pointing at a ui colon slash slash resource. And that resource is an HTML page — usually bundled with its own JavaScript and CSS — that the host renders in place of plain text output.

Lifecycle, step by step

  1. UI preloading — the host can prefetch the resource before the tool is even called.
  2. Resource fetch — host fetches the HTML bundle from the server.
  3. Sandboxed rendering — rendered inside a sandboxed iframe in the conversation.
  4. Bidirectional communication — app and host talk over JSON-RPC on postMessage.
Four steps. First, UI preloading — because the tool description already names the resource, the host can prefetch it before the tool is even called. Second, resource fetch — the host pulls the HTML bundle from the server. Third, sandboxed rendering — that HTML goes into a sandboxed iframe right in the conversation. And fourth, bidirectional communication — the app and the host talk over a JSON-RPC dialect of MCP carried on postMessage.

The full sequence

sequenceDiagram participant User participant Agent participant App participant Server User->>Agent: show me analytics Agent->>Server: tools/call Server-->>Agent: tool result Agent-->>App: result pushed User->>App: user interacts App->>Agent: tools/call Agent->>Server: forwarded Server-->>Agent: fresh data Agent-->>App: fresh data
Here's the general shape, independent of any one example. The user asks something, the agent calls a tool, the server responds, and the agent pushes that result into the app. Then when the user interacts with the app itself, the app can send its own tools call request back through the agent, which forwards it to the server, gets fresh data, and pushes it back. The app never talks to the server directly — the agent, meaning the host, always sits in the middle.

Security model

Iframe sandbox

Can't touch the parent DOM, cookies, local storage, or navigate the parent page.

postMessage only

All communication goes through a channel the host fully controls.

Pre-declared templates

The UI is named ahead of time, so hosts can review, cache, and audit it before anything runs.

Auditable messaging

Every UI-initiated action goes through the same consent path as a direct tool call.
The security model has four pillars. Iframe sandboxing — the app can't touch the parent page, read cookies, or navigate away. postMessage as the only channel — the host controls what capabilities the app can reach. Pre-declared templates — because the UI resource is named ahead of time in the tool description, hosts can review and cache it before anything executes. And auditable messaging — every action the UI initiates rides the same consent and audit path as a direct tool call. No special back door.

Framework support

No required framework — it's standard web primitives: HTML, JS, CSS, and postMessage. Two paths depending on which side you're building:

  • Building a host? Use @mcp-ui/client, or the official App Bridge module for iframe rendering, message passing, and policy enforcement.
  • Building a server/app? @modelcontextprotocol/ext-apps provides a convenience App class — not mandatory, just a wrapper.
  • Starter templates exist for React, Vue, Svelte, Preact, Solid, and vanilla JS.
No framework is required — under the hood it's just HTML, JavaScript, CSS, and postMessage. If you're building a host, the ext-apps SDK includes an App Bridge module that handles iframe rendering and message passing for you, or there's the community mcp-ui client package. If you're building a server or app, there's a convenience App class, but it's a wrapper, not a requirement — you can implement the raw protocol yourself.
