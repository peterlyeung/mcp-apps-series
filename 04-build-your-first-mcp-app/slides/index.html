MCP Apps in Claude Desktop

How the connection, the response, and the render actually work

Episode 04 — mcp-apps-series

In the last episode we built a working MCP App from scratch — a dice roller that renders as an interactive card inside Claude Desktop. In this episode, we're going to slow down and look at exactly what's happening under the hood: how Claude Desktop finds and talks to the server, what comes back over the wire, and how that turns into a clickable UI in the chat.

An MCP server can live in two places

Before we look at our demo, one thing worth being clear on: our dice server runs on your laptop. That's one option, not the only one.

flowchart LR subgraph Local["Local transport"] HD1["Claude Desktop"] S1["MCP server\non your machine"] HD1 <-- "spawns as a subprocess\nstdin/stdout JSON-RPC" --> S1 end subgraph Remote["Remote transport"] HD2["Claude Desktop\nor claude.ai"] S2["MCP server\non someone else's infrastructure"] HD2 <-- "network call over HTTPS\nStreamable HTTP + OAuth" --> S2 end
MCP actually supports two transports. Stdio is local: Claude Desktop launches the server as a child process on your machine and talks to it over standard input and output. That's what we built. But there's also Streamable HTTP: a remote MCP server running on someone else's infrastructure — a company's cloud, a SaaS product — that Claude Desktop talks to over the network instead, usually authenticating with OAuth. In Claude Desktop, local servers go in the config file; remote ones get added as Custom Connectors in Settings. Everything we're about to walk through — the tool declaring a UI resource, the sandboxed iframe, the handshake — works identically either way. Local versus remote is only about how Claude Desktop reaches the server, not about how the UI renders.

Two separate channels: Claude Desktop <-> server.py over stdio, and Claude Desktop <-> iframe over postMessage. The iframe never talks to server.py directly.

Claude Desktop doesn't connect to our server over a network — it launches server.py as a child process and talks to it over stdin and stdout. Every message in both directions is one line of JSON-RPC. And notice there are two separate channels here: Claude Desktop to server-dot-py over stdio, and Claude Desktop to the sandboxed iframe over postMessage. The iframe never talks to our Python server directly — the host always sits in the middle and mediates everything. That's the security boundary.

What the server responds with

Client sendsServer responds with
initializeprotocolVersion, capabilities, serverInfo
tools/listroll_dice tool, with _meta.ui.resourceUri
tools/calltext content + structuredContent (the real data)
resources/listone resource, mimeType: text/html;profile=mcp-app
resources/readthe full HTML/JS page for the UI

The _meta.ui.resourceUri field and the mcp-app mime type are the two signals that tell Claude Desktop "render this as a UI," not just text.

Five request types, five response shapes. The two that matter most: when the host asks for the tool list, our roll_dice tool includes a meta dot ui dot resourceUri field — that's the line that says "this tool has a UI, go fetch it." And when the host reads that resource, it comes back with a mime type of text-html profile mcp-app. Those two signals together are what tell Claude Desktop to render this as an interactive card instead of just printing text.

How it renders: tool call to clickable card

sequenceDiagram participant User participant Host as Claude Desktop participant Server as server.py participant App as MCP App iframe User->>Host: "roll a d20" Host->>Server: "tools/call roll_dice, sides=20" Server-->>Host: "content + structuredContent, sides=20 value=19" Note over Host: tool definition said this tool has a ui resource Host->>Server: "resources/read ui dice-server roller" Server-->>Host: "HTML/JS page, mimeType text/html mcp-app profile" Host->>App: "render HTML in sandboxed iframe" App->>Host: "ui/initialize, sent over postMessage" Host-->>App: "hostInfo, hostContext - theme, size" App->>Host: "ui/notifications/initialized" Host-->>App: "ui/notifications/tool-result, structuredContent" Note over App: renders 19, rolled a 19 on a d20
Here's the full lifecycle. The model decides to call roll_dice. The server responds with structured content. The host notices the tool declared a UI resource, fetches it, and renders it in a sandboxed iframe. The iframe then initializes itself by sending a ui-initialize message to the host over postMessage, the host responds with context like the current theme, the iframe confirms it's ready, and then the host pushes the actual roll result into the iframe as a notification. Notice the app didn't ask for that result — the host just delivered it, because it already knew this was the tool call that triggered the render.

The part that makes it interactive

sequenceDiagram participant User participant Host as Claude Desktop participant Server as server.py participant App as MCP App iframe User->>App: clicks Roll Again App->>Host: "tools/call roll_dice, sides=20 - own request id" Host->>Server: "tools/call roll_dice, sides=20" Server-->>Host: "content + structuredContent, new value" Host-->>App: "response matched by request id" Note over App: re-renders with new value, no model round trip
  • The first roll is model-initiated — pushed to the app as a notification.
  • "Roll Again" is app-initiated — the iframe sends its own tools/call request.
  • The model is not involved in the second leg at all.
This is the half of the story that's easy to miss. Clicking Roll Again doesn't go back to the model at all. The iframe itself sends a tools-call request, with its own request id, straight to the host. The host forwards it to server-dot-py, gets a fresh roll, and routes the response back to the iframe by matching that id. That's what turns this from a one-shot snapshot into an actual interactive app — the UI can keep calling tools on its own.

Recap

  • MCP servers can be local (stdio, a subprocess) or remote (Streamable HTTP, a Custom Connector) — MCP Apps rendering works the same either way.
  • A tool opts into a UI via _meta.ui.resourceUri; the resource is served with mime type text/html;profile=mcp-app.
  • The UI renders in a sandboxed iframe that only talks to the host, never the server directly.
  • Two distinct round trips: model-initiated (pushed as a notification) and app-initiated (the UI calling tools on its own).

So to recap: local versus remote is just about how Claude Desktop reaches the server. What actually makes an MCP App work is a tool declaring a UI resource, that resource rendering in a sandboxed iframe, and two kinds of tool calls — one pushed by the host when the model calls the tool, and one the app can trigger itself.
